# Agents 使用说明

## 1. 文档目的
本目录用于沉淀「大学城美食地图」项目中的 AI 协作体系，覆盖角色定义、提示词模板、执行工作流、复用技能和工具规范。

目标不是“写几个提示词”，而是建立一套可复现、可评审、可迭代的 AI 开发流程。

## 2. 目录结构

- `prompts/`
	- 通用提示词模板，解决“怎么问AI”问题。
- `workflows/`
	- 分阶段执行路径，解决“先做什么后做什么”问题。
- `skills/`
	- 可复用能力模块，解决“常见任务怎么标准化”问题。
- `tools/`
	- 工具调用规范，解决“在哪些边界内安全高效执行”问题。

## 3. 推荐使用方式

### 3.1 单任务执行（例如新建一个页面）

1. 先用 `prompts/system-role.md` 固定上下文。
2. 选择对应 `workflows/*.md`（例如页面脚手架）。
3. 按需加载 `skills/*.md`（例如响应式布局、地图集成）。
4. 使用 `tools/*.md` 约束工具调用和输出格式。

### 3.2 复杂任务执行（例如从文档到可演示闭环）

1. 拆分阶段：结构搭建 -> 页面实现 -> 数据接入 -> 体验打磨 -> 部署。
2. 每阶段绑定一个workflow，并记录输入和产出。
3. 每阶段完成后执行 `prompts/review-checklist.md` 自检。

## 4. 产出物标准

每次 Agent 执行都应产出以下最小信息：

1. 任务目标：一句话说明这次要完成什么。
2. 输入上下文：所用文档、数据、约束。
3. 执行动作：具体变更的文件与结构。
4. 结果摘要：完成项、未完成项、风险项。
5. 下一步建议：可直接执行的后续动作。

## 5. 质量控制

### 5.1 强制检查项

1. 是否偏离“学生决策效率”核心目标。
2. 是否出现只堆功能不讲价值的输出。
3. 是否保持文档路径、代码路径一致。
4. 是否有可验证的验收标准。

### 5.2 常见反模式

- 反模式1：Prompt过泛，产出空洞。
	- 修正：增加场景约束和输出结构约束。
- 反模式2：跳过工作流直接写代码。
	- 修正：先定义阶段目标与验收标准。
- 反模式3：只给“建议”，不落地到文件。
	- 修正：要求输出明确文件路径与变更点。

## 6. 与仓库文档联动

建议按以下顺序联动：

1. `docs/product/*` 定义“做什么”。
2. `agents/*` 定义“怎么高效做”。
3. `src/*` 进行实现。
4. `docs/deliverables/*` 回收演示和复盘。

## 7. 版本维护建议

- 每次大迭代后更新对应 workflow 和 skill。
- 若某模板失效，记录“失效原因+替代模板”。
- 不删除历史经验，采用“新增版本段落”方式演进。
