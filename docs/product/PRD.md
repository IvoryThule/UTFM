

# "吃什么" — 大学城美食地图 产品需求文档

> **版本**：V1.0 | **作者**：IvoryThule | **日期**：2026-03-
> **产品定位**：面向大学城学生群体的 AI 驱动美食决策助手
> **一句话介绍**：不只是找餐厅，更是帮你在30秒内决定——今天吃什么。

---

## 一、项目背景与机会

### 1.1 市场背景

全国高校大学城模式日趋成熟，3-5所高校聚集区周边通常汇聚200+餐饮商户。以北京海淀学院路大学城为例：

- **学生日均决策3次**："早/中/晚吃什么"是每天必答的高频问题
- **信息极度分散**：大众点评覆盖全城但缺乏学生视角；小红书内容生动但碎片化，无法直接决策；校园论坛时效性差、不可检索
- **学生群体特征鲜明**：价格敏感、效率导向、社交驱动、手机重度用户
- **现有方案均不够好**：没有一个产品是"以大学生为中心"设计的本地美食工具

### 1.2 机会判断

| 维度 | 现状痛点 | 产品机会 |
|------|---------|---------|
| 决策效率 | 每次选餐厅平均花10-15min浏览多个平台 | AI 30秒给出精准推荐，直接可行动 |
| 信息匹配 | 通用平台未区分学生/白领/游客 | 学生专属视角：价格×距离×场景联合筛选 |
| 场景覆盖 | 现有平台按品类组织（火锅/烧烤） | 按真实生活场景组织（一个人吃/聚餐/约会/夜宵） |
| 优惠聚合 | 学生折扣分散在各个渠道和社群 | 一站式学生优惠信息聚合 |
| 本地信任 | 大平台评价以游客/上班族为主 | "同校同学真实评价"建立信任感 |

### 1.3 竞品分析

| 竞品 | 优势 | 不足 | "吃什么"的差异化 |
|------|------|------|----------------|
| **大众点评** | 数据全面、商户覆盖广 | 信息冗杂，缺乏学生化视角，决策成本高 | 学生场景化 + AI 对话式决策 |
| **小红书** | 内容生动、种草力强 | 碎片化、搜索效率低、无法结构化决策 | 结构化数据 + 可直接行动的推荐结果 |
| **校园论坛/微信群** | 真实、本地化 | 信息时效差、不可检索、沉没快 | 系统化沉淀 + 实时更新 + AI检索 |
| **美团外卖** | 下单闭环完善 | 外卖视角，不适合"到店探索"场景 | 到店发现 + 探索乐趣 |

> **核心洞察**：现有产品要么"大而全但不懂学生"，要么"真实但不结构化"。**"吃什么"要做的是：用AI把学生最真实的需求和最本地化的信息高效匹配起来。**

## 二、目标用户

### 2.1 核心用户画像

**主要用户**：大学城在校本科生/研究生（18-26岁）

- 月餐饮消费 800-2000 元，80%的餐次人均在15-40元区间
- 日均打开美食/外卖相关App 2-3次
- 90%以上使用手机浏览，几乎不会用PC
- 强社交属性：朋友/同学推荐的权重 > 算法推荐
- 价格敏感但愿意为"特殊场景"（约会/庆祝）偶尔消费升级

### 2.2 用户细分

| 用户角色 | 占比(预估) | 核心诉求 | 典型使用频率 |
|---------|-----------|---------|------------|
| 🍜 日常觅食型 | 50% | 快速决策"中午/晚上吃什么"，不想动脑 | 每天1-2次 |
| 🎉 社交聚餐型 | 25% | 找到适合多人/约会/请客的餐厅 | 每周1-2次 |
| 🆕 探店尝鲜型 | 15% | 发现新店、打卡种草、不想总吃老三样 | 每周2-3次 |
| 💰 省钱至上型 | 10% | 寻找优惠、高性价比推荐 | 每天1次 |

### 2.3 典型用户故事

#### 👩‍🎓 小林 — "选择困难症重度患者"

- **基本信息**：北京大学大三，女，计算机系
- **月餐饮预算**：1200元，日常人均20-35元
- **痛点**：每天中午11:50下课，站在校门口纠结10分钟最后还是去了"老三样"。打开大众点评翻了10页也没选出来，最后饿得随便进一家
- **期望**：有人直接告诉我"今天就吃这个"
- **使用场景**：中午下课 → 打开"吃什么" → 点"AI帮我选" → 输入"一个人，想吃面，不要太远" → 推荐出张妈妈小面(步行8分钟) → 直接导航出发 → 8分钟后已经坐在店里了

#### 🧑‍🎓 阿凯 — "社团聚餐组织者"

- **基本信息**：清华大学研一，男，社团负责人
- **月餐饮预算**：1800元，聚餐时人均50-70
- **痛点**：每周至少组织一次社团聚餐，每次都要同时考虑"能坐下8个人+人均别太贵+别太远+上次去过的不要"，在大众点评上反复筛选30分钟
- **期望**：告诉App我的条件，直接给我合适的3个选项，我选一个就行
- **使用场景**：周五下午准备周六聚餐 → 打开"吃什么" → "8个人聚餐，人均50以内，离清华近，要有大桌" → 对比3家推荐 → 直接打电话预约

#### 💰 小雨 — "月末生存模式选手"

- **基本信息**：人大大二，女，文学系
- **月餐饮预算**：1000元，月底经常只剩200-300块
- **痛点**：月底想找"15块以内能吃饱"的地方，但大众点评搜"便宜"出来的要么远要么难吃，没有靠谱参考
- **期望**：快速找到"便宜但不难吃"的选项，最好还有学生优惠
- **使用场景**：月底 → 点"穷鬼快乐餐"场景入口 → 看到人大东门8元兰州拉面(4.5星/326位同学推荐) → 收藏进"省钱必备"清单

## 三、产品定位与核心价值

### 3.1 产品定位

> **"吃什么"——大学生专属的AI美食决策助手。**
> 不是另一个大众点评，而是一个"懂你的学长/学姐帮你选餐厅"的体验。

### 3.2 核心价值对比

```
传统路径：想吃什么？→ 打开大众点评 → 搜索 → 翻10页评价 → 纠结 → 算了随便吃
"吃什么"：想吃什么？→ 告诉AI你的需求 → 30秒精准推荐 → 直接出发！
```

### 3.3 产品价值三角

```
              用户价值
            ╱         ╲
     决策效率          发现乐趣
   (30秒选好餐厅)    (发现宝藏新店)
         |              |
         ╲            ╱
         商户价值  ←→  平台价值
    (精准学生获客)    (学生餐饮数据沉淀)
```

### 5.2 场景化入口设计逻辑

场景入口是本产品区别于传统"按品类浏览"的核心设计。设计原则：**按学生最高频的用餐决策场景组织，而非按菜品分类逻辑。**

| 场景 | 对应决策维度 | 为什么选这个场景 | 筛选规则 |
|------|------------|---------------|---------|
| 🍜 一个人吃 | 人数=1，快速出餐，不尴尬 | 最高频场景（学生60%+的用餐是一个人） | 出餐快、有单人座、人均<30 |
| 👫 情侣约会 | 氛围 > 价格，环境好 | 差异化需求明确，传统平台不好筛 | 环境评分高、有双人套餐 |
| 🎉 宿舍聚餐 | 人数多，人均可控，有大桌 | 社交场景刚需，每周发生 | 可坐6人+、有拼桌/包间 |
| 🌙 深夜食堂 | 时段=21:00后，还在营业 | 大学生独特时段，其他平台不做时段筛选 | 营业至22:00后 |
| 💰 穷鬼快乐餐 | 价格<15元，吃饱就行 | 月底高频痛点，真实刚需 | 人均<15、评分>4.0 |

> **说明**：这5个场景不追求分类学上的MECE（互斥穷尽），而追求**"用户一眼看到就知道点哪个"的直觉匹配**。实际使用中一个用户可能在不同时刻切换不同场景。

#### AI 推荐引擎设计（V1 前端实现方案）

由于V1是Demo，推荐引擎采用**前端规则匹配**方式，不依赖后端ML模型或LLM API调用：

**Step 1：意图解析**

从用户输入文本中提取关键维度：

| 维度 | 识别方式 | 示例输入 | 解析结果 |
|------|---------|---------|---------|
| 预算 | 数字 + "元/块/以内/左右" | "人均30以内" | budget ≤ 30 |
| 品类 | 品类关键词库匹配 | "想吃火锅" | category = "火锅" |
| 人数 | 数字 + "人/个人" | "4个人" | partySize = 4 |
| 距离 | 学校名 / "近/远" | "离北大近" | sortByDistance("PKU") |
| 场景 | 场景词库匹配 | "约会/聚餐/夜宵" | scene = "date" |
| 口味 | 口味关键词库 | "想吃辣的" | taste = "spicy" |
| 排除 | "不要/不想/别" + 品类 | "不要太辣" | exclude: spicy |

```typescript
// 意图解析核心逻辑（简化示例）
function parseIntent(input: string): UserIntent {
  const intent: UserIntent = {};

  // 预算识别
  const priceMatch = input.match(/(\d+)\s*(元|块|以内|左右)/);
  if (priceMatch) intent.maxPrice = parseInt(priceMatch[1]);

  // 品类识别
  const categories = ['火锅','烧烤','面','饺子','奶茶','日料','韩餐','西餐','炒菜'];
  categories.forEach(cat => {
    if (input.includes(cat)) intent.category = cat;
  });

  // 场景识别
  if (input.includes('一个人')) intent.scene = 'solo';
  if (input.match(/约会|浪漫|纪念日/)) intent.scene = 'date';
  if (input.match(/聚餐|聚会|多人/)) intent.scene = 'group';
  if (input.match(/夜宵|深夜|宵夜/)) intent.scene = 'late-night';

  // 更多维度解析...
  return intent;
}
```

**Step 2：筛选过滤**

从本地餐厅数据中过滤符合所有条件的商户。

**Step 3：排序打分**

对筛选结果按以下优先级排序：
1. **条件匹配度**（匹配的维度越多，得分越高）
2. **同校学生推荐数**（社交证明，权重最高）
3. **综合评分**（评分高的优先）
4. **距离**（近的优先，距离衰减）

```typescript
function scoreRestaurant(r: Restaurant, intent: UserIntent, university: string): number {
  let score = 0;

  // 条件匹配度（每匹配一个维度+20分）
  if (intent.maxPrice && r.avgPrice <= intent.maxPrice) score += 20;
  if (intent.category && r.category.includes(intent.category)) score += 20;
  // ...更多维度

  // 同校推荐数（归一化到0-30分）
  score += Math.min(30, r.studentCount / 100);

  // 评分（0-20分）
  score += r.rating * 4;

  // 距离衰减（步行15min以内，越近分越高）
  const walkMin = r.walkingDistance[university] || 15;
  score += Math.max(0, (15 - walkMin)) * 2;

  return score;
}
```

**Step 4：结果包装**

为每个推荐自动生成一句"推荐理由"，突出最匹配的维度：

```typescript
function generateReason(r: Restaurant, intent: UserIntent): string {
  if (intent.maxPrice && r.avgPrice <= intent.maxPrice * 0.7)
    return `人均仅${r.avgPrice}元，比你的预算还省${intent.maxPrice - r.avgPrice}块`;
  if (r.studentCount > 1000)
    return `${r.studentCount}位同学推荐，校门口的"食堂级"存在`;
  if (r.walkingDistance[currentUniversity] <= 5)
    return `下课走路${r.walkingDistance[currentUniversity]}分钟就到，快过去食堂排队`;
  return `${r.mustOrderDishes[0]}是招牌必点，好评率超90%`;
}
```

#### AI 交互体验细节

| 细节 | 实现方式 | 为什么重要 |
|------|---------|-----------|
| 打字机效果 | AI回复文字逐字显示，20-40ms/字 | 营造AI"思考后回答"的感觉 |
| 思考延迟 | 用户发送后等待800-1200ms再开始回复 | 太快会觉得是假的 |
| 推荐卡片进场 | 从底部滑入+淡入，间隔200ms逐个出现 | 视觉引导，有节奏感 |

| 追问/换一批 | 底部提供"换一批推荐"按钮 | 第一次推荐不满意时有出路 |

| 维度 | 指标 | 目标 | 估算依据 |
|------|------|------|---------|
| 用户规模 | DAU | 500+ | 4校约2万学生，渗透率2.5%即可达到 |
| 用户留存 | 次日留存率 | ≥ 35% | 工具类App行业均值约30%，"吃什么"具备高频+本地化优势，预期略高于均值 |
| AI推荐效果 | 推荐→点击率 | ≥ 50% | 参考AI对话类产品结果点击率40-60%，取中位数 |
| 决策效率 | 从打开到点击"导航/详情"的平均时长 | ≤ 60秒 | 产品核心承诺"30秒决策"的验证 |
| 内容健康 | UGC评价数/日 | 20+ | V1.5上线评价系统后的目标 |

> **说明**：以上为假设性目标，实际需在MVP上线后通过数据回收调整。V1 Demo阶段核心验证的行为假设是："面对'吃什么'这个问题，学生是否愿意用AI推荐替代自主搜索浏览？"

## 十、商业化思考

### 10.1 变现路径（分阶段）

```
阶段一：用户积累（V1 - V2）
  目标：积累学生用户和UGC数据，验证产品价值
  策略：完全免费使用
  收入：无

阶段二：流量变现（V2.5 - V3）
  目标：基于用户流量实现初步变现
  策略：
  ├── 🏪 餐厅入驻推广费（竞价排名/Banner/榜单置顶）
  ├── 🎫 优惠券分发佣金（CPS模式，每核销一张抽成）
  └── 🤝 品牌联名活动（如：瑞幸×"吃什么"开学季）
  预期：单校区月收入 5K-2W

阶段三：生态协同（V3+）
  目标：与美团生态打通，实现规模化
  策略：
  ├── 📱 与美团外卖/团购打通 → 发现→下单闭环
  ├── 🎪 校园美食节IP活动运营
  └── 📊 学生消费数据洞察服务（脱敏后提供给商户）
```

### 10.2 与美团的协同价值

| 协同场景 | 对"吃什么"的价值 | 对美团的价值 |
|---------|---------------|------------|
| 外卖打通 | AI推荐结果可直接跳转美团下单，闭环体验 | 新增学生群体的外卖流量入口 |
| 到店团购 | 餐厅详情页可展示美团团购券，提升转化 | 到店场景的精准导流 |
| 数据协同 | 获取商户基础数据（菜单/价格/评分） | 获取学生群体的美食偏好数据 |
| 品牌营销 | 联合品牌活动带来用户增长 | 学生市场的品牌渗透 |

### 10.3 商业模式画布（简版）

| 要素 | 内容 |
|------|------|
| 客户细分 | C端：大学城在校学生 \| B端：周边餐饮商户 |
| 价值主张 | C端：30秒高效美食决策 \| B端：精准学生客群触达 |
| 渠道通路 | 校园社群裂变、新生群推广、高校合作、社交媒体种草 |
| 核心资源 | AI推荐引擎 + 学生UGC数据 + 本地化商户信息 |
| 成本结构 | 服务器/API成本 + 初期内容运营人力 |

## 十二、风险与应对

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|---------|
| 冷启动缺少内容 | 高 | 新用户进来看到空数据会流失 | V1使用精心编写的种子数据；V1.5通过"评价送奖励"激励UGC |
| AI推荐不够准 | 中 | 用户对AI能力失望后不再使用 | V1基于规则保证基础匹配度；收集用户反馈迭代；提供"换一批"兜底 |
| 用户生命周期短(毕业) | 中 | 每年流失25%用户 | 利用"新生入学"节点集中获客；高年级→低年级的口碑传递 |
| 竞争对手模仿 | 低 | 大众点评/美团推出类似功能 | 深耕UGC内容壁垒和校园社群关系；速度是最好的壁垒 |
