# 技术选型说明（Tech Stack）

## 1. 选型目标

- 快速完成可演示Demo，优先保证产品结构和交互流程。
- 低实现复杂度，便于在测评时间内稳定交付。
- 支持后续平滑扩展到真实数据与地图服务。

## 2. 推荐技术栈

### 2.1 前端

- 框架：Next.js（App Router）
- 语言：TypeScript
- 样式：Tailwind CSS
- 状态管理：React Hooks（V1轻量，不引入复杂状态库）

选择理由：

- 页面路由组织清晰，适合多页面演示。
- TypeScript有利于接口与数据结构对齐。
- Tailwind可快速迭代移动端页面样式。

### 2.2 数据与接口

- V1数据源：本地Mock数据（src/data）
- 接口形态：Next.js Route Handler或轻量Node接口
- 推荐逻辑：规则引擎优先（预算、距离、场景、热度）

选择理由：

- 避免重后端开发，聚焦产品验证。
- 规则可解释，便于面试说明推荐依据。

### 2.3 地图能力

- V1：静态/半静态地图组件（可模拟点位）
- V2：接入高德或Mapbox SDK

选择理由：

- V1先验证体验与流程，不被地图API调试耗时拖累。

## 3. 工程组织建议

- 页面路由：src/app
- 业务组件：src/components/business
- 数据层：src/data
- 类型定义：src/types
- 工具函数：src/lib

## 4. 非功能需求

- 移动端优先（主流手机宽度适配）
- 首屏加载尽量轻量（避免大图与重依赖）
- 错误可恢复（接口失败有重试与兜底文案）

## 5. 风险与替代方案

| 风险 | 影响 | 替代方案 |
|---|---|---|
| 地图SDK接入耗时 | 影响交付节奏 | V1使用简化地图视图，V2再接真地图 |
| 推荐结果不稳定 | 用户不信任 | 规则兜底 + 理由可解释 |
| 样式打磨时间不足 | 演示质感下降 | 先保主链路可用，再做视觉增强 |

## 6. 结论
本项目采用“轻后端+强前端展示+可解释推荐规则”的方案，最符合测评目标：
在有限时间内最大化产品思维与AI协作能力的呈现。
